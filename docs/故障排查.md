# 故障排查

> 基于**当前已实现能力**的排障案例。部分指标名为目标设计，尚未全部暴露为 metrics。

## 快速检查清单

| 现象 | 检查项 |
| --- | --- |
| 进程起不来 | `edgeflow-cli validate-config`；日志目录写权限 |
| 无 telemetry | `simulate` 是否为 true；libmodbus 连接是否成功 |
| MQTT 无数据 | `broker_host`/`broker_port`；`mosquitto_sub -t '#'` |
| metrics 为空 | `metrics_path` 目录是否存在、可写 |
| 串口失败 | `/dev/ttyUSB0` 权限、`dialout` 组、`modbus_slave_id` |
| 补传不工作 | SQLite `uploaded=0` 是否有数据；broker 是否恢复 |

## 案例 1：MQTT broker 不可达，本地仍在采集

### 现象

云端收不到数据，但日志仍刷新 telemetry，策略仍可能生成 command。

### 相关 metrics

```text
edgeflow_mqtt_publish_fail_total 增长
edgeflow_storage_pending_count 增长
```

### 根因

broker 不可达。设计上云端失败不影响本地采集与落库，数据以 `uploaded=0` 缓存。

### 处理

1. 检查 broker 地址、端口、防火墙
2. 恢复 broker 后观察 `edgeflow_mqtt_replay_total` 与 pending 下降
3. `edgeflow-cli storage-stats --sqlite <path>`

### 验证

```bash
mosquitto_sub -t 'plant/line1/gw-001/#' -v
sqlite3 /tmp/edgeflow/edgeflow.db "SELECT COUNT(*) FROM telemetry WHERE uploaded=0;"
```

## 案例 2：libmodbus 连接失败

### 现象

日志出现 `libmodbus connect failed` 或 `libmodbus read failed`，ingress 无新数据。

### 根因

- RTU：串口不存在、权限不足、波特率/从站 ID 错误
- TCP：IP/端口错误、防火墙、从站未监听 502

### 处理

1. RTU：`ls -l /dev/ttyUSB0`，确认 `modbus_baud_rate`、`modbus_slave_id`
2. TCP：`nc -zv <host> 502`，确认 `modbus_transport=tcp`
3. 先用 `simulate=true` 验证主链路正常，再切真实设备

## 案例 3：队列丢弃增长

### 现象

`edgeflow_queue_dropped_total` 持续增加。

### 根因

ingress 采集快于 worker 处理，`queue_size` 过小或 worker 被 SQLite/MQTT 阻塞。

### 处理

1. 增大 `queue_size`
2. 适当增大 `poll_interval_ms`
3. 检查 MQTT publish 是否长期失败导致 worker 变慢

## 案例 4：systemd 频繁重启

### 现象

`systemctl status edgeflow` 显示反复 restart。

### 处理

1. `journalctl -u edgeflow -n 100` 看崩溃原因
2. 确认 `WatchdogSec` 与 `watchdog_interval_ms` 匹配（monitor 需周期喂狗）
3. 确认 `NOTIFY_SOCKET` 环境下 `watchdog_notify_ready` 已调用

## 案例 5：配置加载失败

### 现象

启动报 `failed to load config`。

### 处理

```bash
./build/edgeflow-cli validate-config -c configs/gateway.json
```

检查 JSON 语法、cJSON 不支持的注释（勿在 JSON 内写 `//` 注释）。
